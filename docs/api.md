# Agent-Friendly API

The API is the main differentiator of Agent Taskboards. It should let coding
agents manage local task state through stable, scriptable operations instead of
driving the UI. The current implementation only exposes `GET /api/health`; the
rest of this document describes the intended v1 API direction.

## API Principles

- JSON-first: requests and responses use JSON by default.
- Stable IDs: projects, boards, tasks, comments, and activity entries have
  durable IDs suitable for agent memory.
- Canonical writes: create, update, move, and archive operations return the
  canonical resource after the change.
- Explicit transitions: moving a task uses a specific operation instead of
  requiring clients to rewrite a whole board.
- Predictable errors: failures return a consistent shape with an error code,
  message, and optional details.
- Timestamped resources: resources include creation and update timestamps.
- Shell-friendly: responses are deterministic and easy to consume from `curl`,
  `jq`, and small bash helper scripts.
- Local semantics: no account, workspace, or permission concepts are required in
  v1 because the app is single-user.

## Resource Hierarchy

The planned v1 hierarchy is:

```text
/api/projects
/api/projects/:projectId/boards
/api/projects/:projectId/boards/:boardId/tasks
/api/tasks/:taskId/comments
/api/tasks/:taskId/activity
```

The exact route names can evolve during implementation, but the hierarchy and
semantics should remain centered on `project -> board -> task -> comments/activity`.

## Health

`GET /api/health`

Returns service status. The current scaffold returns `{ "ok": true }`. Future
versions may include database and embedding index health.

## Projects

Planned project operations:

- list active projects
- create a project
- read a project by ID
- update project metadata
- archive a project

Project responses should include enough summary information for an agent to pick
the correct scope before operating on boards or tasks.

## Boards

Planned board operations:

- list boards within a project
- create a board within a project
- read a board by ID, including columns and task summaries when requested
- update board metadata and workflow columns
- archive a board

Board reads should support both compact responses for agent selection and fuller
responses for UI rendering.

## Tasks

Planned task operations:

- list tasks within a board
- create a task within a board
- read a task by ID
- update task fields such as title, description, priority, labels, and
  references
- move a task to another column or position
- archive or complete a task

Task movement should be a first-class API operation with explicit source and
destination semantics. Agents should be able to say "move task X to blocked"
without reconstructing the board state.

## Comments and Activity

Planned comment and activity operations:

- append a task comment
- list task comments
- list task activity
- fetch combined task context for agent handoff or memory retrieval

Comments should support human and agent authorship metadata. Activity entries
should be generated by the system for important changes such as create, move,
update, archive, and complete.

## Search

Search is a first-class v1 API area. It should support querying across boards,
tasks, and task comments.

Planned search modes:

- text search for exact or keyword-style lookup
- semantic search backed by local embeddings
- scoped search within a project, board, or task
- filters for active versus archived content

Search results should identify the matching object type, ID, parent context,
score or rank, and a compact snippet when possible. Agents should be able to use
search as long-term memory before starting or continuing work.

## Maintenance

Planned maintenance operations:

- inspect storage and embedding index health
- reindex embeddings for projects, boards, tasks, and comments
- purge old archived data when explicitly requested
- compact or vacuum local storage when safe

Maintenance APIs should be available to both the UI and trusted local agents.
Potentially destructive operations should require explicit confirmation fields
or similarly deliberate request shapes.

## Error Shape

Errors should use a consistent JSON shape:

```json
{
  "error": {
    "code": "not_found",
    "message": "Task not found",
    "details": {}
  }
}
```

HTTP status codes should match the failure class, while the JSON code should be
stable enough for agents to branch on.
