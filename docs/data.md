# Data Storage And API Data Types

This document covers lower-level data handling details for Agent Taskboards.
Core project-management concepts are documented in `docs/tasks-and-boards.md`.

## Storage Overview

SQLite is the source of truth for application data. The default database path is:

```text
/data/taskboards.sqlite
```

The host-visible runtime directory is `data/`, mounted into the container at
`/data`. The API uses Drizzle ORM with `better-sqlite3` and applies generated SQL
migrations from `drizzle/` at startup.

The database client enables:

- SQLite foreign keys
- WAL journal mode
- a busy timeout for concurrent local access

## Runtime Directories

Runtime directories follow the Docker Compose mounts:

- `/data`: durable application data, including the SQLite database
- `/uploads`: durable uploaded or imported files for future attachment flows
- `/tmp/taskboards`: scratch data that can be discarded

Durable user or agent data should not be stored in `/tmp/taskboards`.

## IDs

Persisted resources use durable text IDs. Most resources use IDs generated by
`nanoid` unless an insert explicitly provides an ID. New task IDs are generated
from the task title by taking a sanitized, lowercase, hyphenated prefix from the
first four alphanumeric words and appending a 6-character lowercase
alphanumeric suffix. The task prefix uses crude dependency-free transliteration:
extended Latin diacritics are folded with Unicode normalization and Cyrillic
characters are mapped to approximate Latin equivalents. Latin letters that do
not decompose cleanly, such as `ß`, `œ`, `ø`, `đ`, and `ł`, use a small explicit
map. Task IDs are generated only when the task is created; later title changes
do not rename the task or any child records.

Resources with stable IDs include:

- projects
- boards
- board columns
- tasks
- task comments
- task activity entries
- search documents

IDs are API-facing and suitable for agent memory, scripts, comments, and
handoffs.

## Timestamps

SQLite stores timestamps as integer milliseconds since the Unix epoch. The API
serializes timestamp fields as ISO 8601 strings or `null`.

Common timestamp fields:

- `createdAt`: set at insert time
- `updatedAt`: updated by Drizzle when mutable records change through Drizzle
- `archivedAt`: nullable archive marker
- `completedAt`: nullable task completion marker
- `embeddedAt`: nullable search-document embedding marker

Append-only records such as comments and activity have `createdAt` but no
`updatedAt`.

## JSON Fields

Open-ended data uses SQLite text columns with JSON encoding and Drizzle JSON
mode.

JSON object fields default to `{}`:

- project `metadata`
- board `metadata`
- task `metadata`
- comment `metadata`
- activity `data`
- search document `metadata`

JSON array fields default to `[]`:

- task `labels`
- task `externalReferences`

The API accepts JSON objects for `metadata` and JSON arrays for
`externalReferences`. Task `labels` are currently arrays of non-empty strings.

## Nullability And Optional Inputs

Optional text fields are represented as `null` when absent in API responses.
Create and update request bodies may omit optional fields. Fields that accept
null include:

- project `description`
- project `repositoryPath`
- project `defaultBranch`
- board `description`
- task `description`
- comment `authorName`
- comment `authorRef`
- activity `actorName`
- activity `actorRef`

PATCH endpoints reject empty request bodies. Unknown or invalid enum values are
rejected with `invalid_request`.

## Enums

Supported task priorities:

```text
low, normal, high, urgent
```

Supported actor and author types:

```text
human, agent, system
```

Search document source types:

```text
project, board, task, comment, activity
```

Embedding statuses:

```text
pending, indexed, stale, error
```

Search documents and embedding status fields exist in the schema for planned
search and semantic memory work. They are not yet part of the starter API.

## Parent IDs And Foreign Keys

The database stores parent IDs redundantly on task-adjacent records for easy
scoping and search:

- tasks store `projectId`, `boardId`, and `columnId`
- comments store `projectId`, `boardId`, and `taskId`
- activity entries store `projectId`, `boardId`, and `taskId`
- search documents may store `projectId`, `boardId`, and `taskId`

Foreign keys cascade on hard deletion. Normal API flows archive projects,
boards, and tasks instead of deleting them, so cascades are mainly a maintenance
and schema-safety concern.

## Active Versus Archived Rows

Projects, boards, and tasks use nullable `archivedAt` fields. A row is active
when `archivedAt` is `null`.

Starter API reads exclude archived projects, boards, and tasks unless the
endpoint supports `includeArchived=true`. Comments and activity do not have
their own archive state; they remain attached to their task.

## Ordering

Board columns are ordered by `position`.

Tasks are positioned within a board column using `position`. Task movement
renumbers affected active tasks in the source and destination columns inside a
database transaction. Archived tasks are excluded from movement reordering.

## Current Tables

The current schema contains:

- `projects`
- `boards`
- `board_columns`
- `tasks`
- `task_comments`
- `task_activity`
- `search_documents`
- `schema_migrations`

`schema_migrations` is maintained by the runtime migration runner and records
applied SQL files and checksums.
